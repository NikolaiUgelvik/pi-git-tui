## Review
- Correct: The service/controller extraction is broadly coherent. `git.ts` now only re-exports focused service modules, `git-extras.ts` keeps compatibility exports, and the largest non-test source file is `src/viewer-core.ts` at 427 lines, below the 500-line Biome limit. `npm test`, `npm run typecheck`, and `npm run lint` all pass.
- Correct: The UI boundary is mostly preserved: overlay controllers own local list/search/render state, while viewer classes still perform git side effects and document/cwd updates. This matches the documented boundary in `docs/architecture.md:38-41`.
- Blocker (medium): Async side-effect callbacks are now fired from controllers without `await`/`.catch`, which drops the old viewer-level error handling contract. For stash operations this is a concrete integration risk: `StashPickerController` calls async callbacks directly from sync input handling (`src/stash-picker-controller.ts:156-162`, `src/stash-picker-controller.ts:166-173`), while `runStashCurrent`, `runStashPop`, and `runStashDrop` do additional `listStashes(...)` work after the caught operation path (`src/viewer-stash-picker.ts:102-110`, `src/viewer-stash-picker.ts:125-147`). If that refresh fails, the returned promise rejects without `showAsyncError`, unlike the pre-extraction handlers that wrapped these calls with `.catch(...)`. Smallest safe fix: wrap callbacks at the viewer/controller boundary, e.g. pass `onStashCurrent: () => this.runStashCurrent().catch((error) => this.showAsyncError(error))` (and apply the same pattern to all async controller callbacks), or add an `onError` callback and have controllers `void callback().catch(onError)`.
- Note (low): Overlay chrome/layout logic is duplicated across the extracted controllers. The same `overlayWidth`, `maxItems`, `row`, and `border` setup appears in `src/command-menu-controller.ts:91-103`, `src/commit-picker-controller.ts:103-115`, `src/branch-picker-controller.ts:133-145`, `src/stash-picker-controller.ts:178-190`, and `src/worktree-picker-controller.ts:116-128`. This is not a functional regression, but it weakens the extraction by leaving UI geometry drift-prone. Smallest safe fix: extract a small shared overlay layout/chrome helper used by the controllers.
- Note (low): Some git plumbing remains duplicated across focused services: staged/untracked listing is implemented in both `src/git-diff-service.ts:48-61` and `src/git-index-service.ts:5-10`, `src/git-index-service.ts:51-55`; cached diff argument construction also appears in `src/git-index-service.ts:130-148` while `src/git-diff-service.ts:177-186` owns the shared base diff args. Smallest safe fix: centralize these helpers in one git status/index utility module or export them from a single service to avoid future behavior drift.
- Note: Requested `plan.md` and `progress.md` were not present in the repository, so this review is based on the current unstaged tracked/untracked worktree and test results.
